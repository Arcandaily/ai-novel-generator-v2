# 边缘情况与异常处理规则 v3.23

## 核心理念
真实使用中会遇到各种边缘情况和异常，引擎需要有明确的处理规则，以保证系统的鲁棒性和用户体验。

## 1. 常见边缘情况与处理规则

### 1.1 主要人物死亡或永久离场
- 允许，但必须有充分的因果逻辑和情感/主题准备。
- 操作建议：提前预测风险、及时更新状态和关系、记录为重要记忆。

### 1.2 用户要求中途大幅改变题材/主题
- 允许，但需评估影响范围并给出合理过渡方案。
- 操作建议：评估冲突程度、建议分阶段过渡或重新初始化部分状态、记录变更。

### 1.3 极长篇故事的一致性维护
- 需要更严格的机制。
- 操作建议：定期全量一致性检查、加强并行事件追踪、重要节点创建快照、长期记忆定期整理。

### 1.4 用户强行要求不合理/低质量发展
- 尊重用户，但需进行风险提示和替代方案建议。
- 操作建议：通过 Self-Critique 评估影响、给出风险说明和备选方案、记录为“用户强制分支”。

### 1.5 重大时间跳跃 / 时间跳过
- 允许，但必须合理更新状态并处理信息差。
- 操作建议：跳跃前创建快照、合理更新人物和世界状态、处理信息差、避免无意义跳跃。

### 1.6 信息不对称与误会链
- 允许，但需控制复杂度和可追溯性。
- 操作建议：记录重要信息差、避免过度复杂的误会链、误会解除需有合理触发和情感冲击力。

### 1.7 世界模型规则中途变化
- 谨慎，需有充分理由和过渡。
- 操作建议：通过客观世界逻辑自然引入、更新受影响状态、避免随意修改核心规则。

### 1.8 记忆冲突或矛盾
- 优先保证当前故事的一致性。
- 操作建议：以当前故事内记忆为准、跨故事经验仅作参考、记录冲突并复盘。

## 2. 异常处理通用流程

1. 检测异常
2. 评估影响范围和严重程度
3. 按优先级尝试自动解决
4. 无法自动解决时触发用户介入
5. 执行方案并更新状态、记忆
6. 记录处理过程

## 3. 实施建议

- Logical Deduction Engine 和 Self-Critique 应主动识别潜在边缘情况。
- 重要异常处理过程应记录在状态和记忆中。
- 可根据题材预设常见边缘情况的默认处理倾向。
